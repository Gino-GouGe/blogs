## What is Raft?
Raft is a concensus algorithm for managing replicated state machines. The key compositions of Raft are Leader Election, Log Replication and Safety. Safety puts additional constrains on Leader Election and Log Replication to guarantee the elected leader preserves all the committed states. Other than algorithm correctness and efficiency, one of the goals of Raft is understandability. i.e. to facilitate the development of intuitions that are esential to system builders. System builders can not only understand how it works but why it works to adjust/improve the algorithm for practical usecases.

A basic concensus algorithm requires only Leader Election and Log Replication. This blog is on Leader Election w\o Safety.

## Leader Election Algorithm
Raft algorithm operates on a cluster of odd numbered servers, e.g. a cluster of 3/5/7. 5 is a typical number and can tolerate 2 failures. The state machine and transition is depicted below (See ref for Raft paper)

<img src="{{ site.baseurl }}/assets/images/state_machine.png" alt="State Machine Transition">

1. Every server can be in 3 states, "follower", "candidate", "leader"。 They start at "follower" state and wait for Election Timeout
2. After Randomized Election Timeout, one server switches state to "candidate", bumps-up its term, votes for itself and sends RequestVote RPC to other servers in the cluster
3. Candidate, the sender collects "# of votes" and "# of requests sent" and can run into 3 scenarios below
    * Gets majority votes (of a full cluster), steps up to "leader" and send HeartBeat (empty AppendEntries) immediately to establish authority
    * Discover a higher term, update term to higher term and step down to follower
      * Note: The primary goad to update term to higher term upon discovery is to bring candidate that can be elected as leader, up to the matching term with partitioned candidate with in-complete logs, where the candidate just keeps bumping up it term over and over again. It will make more sense after Safety. For example,
	    1. S0 is elected as leader and sending HeartBeat
        2. S2 is partitioned, doesn't receive HeartBeat, goes into leader election process
  		3. S2 never successfully elected as a leader because of no majority votes from S0 and S1. And it keeps triggering leader election over and over again with term bumping up.
  		4. After some time the replicated logs are [0,1,2] (only term), [0,1,2] and [] repectively on S0, S1, S2. S0 is partitioned and new leader should be elected between S1 and S2. In this case, S1 should be the only server elected because it has the complete logs. However S1 is on term 1 and S2 can be on term 100 and leader election will always fail. So need to bring S1 on term 1 up to 100 immediately.
    * Split vote -> Start Leader Election over again
	  * Note: To prevent constant split vote, a randomized Election Timeout is employed for each server
4. Leader needs to send periodic HeartBeat to followers to maintain its authority
5. Candidate would start a new Leader Election upon election timeout, the pending Election timeout is discarded or ignored.

## Implementation
### Randomized Election Timeout and Periodic HeartBeat (Bullet point 1, 2, 5)
```
for {
    switch state {
    case Leader:
        go sendAppendEntries()
    case Follower:
        fallthrough
    case Candidate:
        if ElectionTimeout {
            go attemptLeaderElection()
        }
    }

    pause_random_amount_of_time()
}
```

### RequestVoteRPC (Bullet point 3, 4)
Note:
* There is room for optimization where a candidate switch to follower, it should suppress unnecessary RequestVote RPCs. But considering candidate sends RequestVote RPC simultaneously in separate threads, this performance improvement is doubtful.
* When reply, server's currentTerm should return to candidate so candidate can discover there could be a higher term and step-down to follower. Additional constrains would be added in Safety to determine whether a candidate has complete logs and can be elected
```
// Sender
func () attemptLeaderElection() {
	state = Candidate
	currentTerm += 1
	votedFor = self
	lastHeartbeatTime = time.Now()
	votesReceivedCnt = 1
	sendRequestCnt = 1

	for follower in follwers {
		go func(server int, term int) {
			ok := sendRequestVote(server, &args, &reply)
			if ok && reply.Term > term {
				currentTerm = reply.Term
				votedFor = -1
				state = Follower
			}

			if ok && reply.VoteGranted {
				votesReceivedCnt += 1
			}
			sendRequestCnt += 1
		}(follower, currentTerm)
	}

	waitUntilEither(votesReceivedCnt >= len(followers)/2 or sendRequestCnt == len(followers))
	if votesReceivedCnt >= len(follower)/2+1 {
		state = Leader
		go sendAppendEntries()
	}
}
// Receiver
func (rf *Raft) RequestVote(args *RequestVoteArgs, reply *RequestVoteReply) {
	if args.Term < rf.currentTerm {
		rejectVote()
		return
	}
	/** Additional Safety constrains to determine whether a candidate can be elected **/
	if args.Term > rf.currentTerm {
		approveVote()
		return
	}

	if args.Term == rf.currentTerm {
		approveVote() if neverVoted or votedFor == args.CandidateId
		else rejectVote()
		return
	}
}
```
